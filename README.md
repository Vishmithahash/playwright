## Overview

This project demonstrates Playwright end-to-end tests. Tests live in the `tests/` folder. The repository includes `server.js` used as a simple web server to run against.

## Prerequisites

- Node.js (16+ recommended)
- npm (or yarn)
- Git (optional)

## Install

1. Install project dependencies:

```
npm install
```

2. Install Playwright browsers (if not installed by postinstall):

```
npx playwright install
```

Note: `playwright` package installs browser binaries when `npx playwright install` runs. Many projects include this in `postinstall`.

## Start the app / webServer

You can start the backend manually or let Playwright start it using the `webServer` option in `playwright.config.js`.

- Manual start (in one terminal):

```
npm start
```

- Let Playwright start it (configured in `playwright.config.js`): Playwright will run the `webServer.command` before tests and stop it after tests (or reuse existing server if configured).

## Run tests

- Run full test suite (headless):

```
npx playwright test
```

- Run tests with headed browsers (see what happens in the actual browser window):

```
npx playwright test --headed
```

- Run a single test file:

```
npx playwright test tests/example.spec.js
```

- Run tests for a specific project/browser (e.g., chromium):

```
npx playwright test --project=chromium
```

- Run tests matching a title (grep):

```
npx playwright test -g "should do something"
```

- Run with debug (Playwright Inspector):

```
npx playwright test --debug
```

## Reports, Traces, Screenshots and Videos

- Open the HTML report generated by the `html` reporter (default here):

```
npx playwright show-report
```

- Collect traces to inspect failing runs. If trace collection is enabled (for example `trace: 'on-first-retry'`), after the test run open the trace file:

```
npx playwright show-trace <path-to-trace.zip-or-trace-folder>
```

- Screenshots and videos: Configure `playwright.config.js` to save `screenshot` and `video` for failures or all runs. Example in config:

```js
use: {
  video: 'retain-on-failure',
  screenshot: 'only-on-failure',
  trace: 'on-first-retry'
}
```

## Debugging failing tests

- Re-run failed tests with `--retries=0` and `--debug` to enter the Playwright inspector.
- Use `trace` to collect a detailed trace; open with `npx playwright show-trace`.
- Add `console.log()` in tests where helpful; Playwright shows test console output.

## Key configuration notes (playwright.config.js)

- `testDir`: where tests live (this repo uses `./tests`).
- `projects`: define browsers/devices to run (chromium, firefox, webkit, mobile emulation).
- `webServer`: start your backend before tests; example:

```js
webServer: {
  command: 'node server.js',
  url: 'http://localhost:3000',
  reuseExistingServer: !process.env.CI,
}
```

- `use`: shared settings for tests: `baseURL`, `trace`, `screenshot`, `video`, timeouts, and `storageState` for auth fixtures.

- Retry and worker settings: set `retries` for flaky tests in CI and `workers` to control parallelism.

## Writing tests — conventions and best practices

- Organize tests in `tests/` by feature or page.
- Prefer `test.describe()` blocks to group related tests.
- Use `page.locator()` (Locators) instead of deprecated `page.$` selectors; locators are resilient and auto-waiting.
- Avoid brittle selectors (avoid relying on text that changes); use `data-testid` or stable attributes when possible.
- Keep tests deterministic: reset state between tests, use fixtures for setup/teardown.
- Use `test.beforeEach()` and `test.afterEach()` for per-test setup/cleanup.
- For authentication, use `storageState` or API login helpers to avoid UI logins in every test.

Example test structure:

```
tests/
  login.spec.js
  dashboard.spec.js
```

Example snippet using locators:

```js
test('user can login', async ({ page }) => {
  await page.goto('/login');
  await page.locator('[data-testid="email"]').fill('user@example.com');
  await page.locator('[data-testid="password"]').fill('password');
  await page.locator('button[type="submit"]').click();
  await expect(page).toHaveURL('/dashboard');
});
```

### Assertions (Playwright `expect`)

- Playwright provides an `expect` assertion library tightly integrated with locators and test fixtures. Use `expect` to wait for conditions automatically (it retries until timeout).
- Common assertions:
  - `await expect(locator).toBeVisible()` — element visible
  - `await expect(locator).toHaveText('text')` — exact text match (or use `toContainText`)
  - `await expect(page).toHaveURL('/dashboard')` — URL assertion
  - `await expect(response).toBeOK()` / `expect(response.status()).toBe(200)` for API responses
  - `await expect(locator).toHaveCount(n)` — number of matched nodes
- Timeouts: pass an options object to override default timeout: `await expect(locator).toBeVisible({ timeout: 5000 })`.
- Use descriptive messages and avoid mixing test logic into assertions. Prefer multiple small assertions over one large combined check for clearer failures.

### Setup / Teardown (`beforeEach` / `afterEach` / fixtures)

- `test.beforeEach()` and `test.afterEach()` run per test to prepare and clean state. Use them for repetitive setup like navigating to a base URL or creating test data.
- Example:

```js
import { test } from '@playwright/test';

test.beforeEach(async ({ page }) => {
  await page.goto('/');
  // optionally reset app state via API
});

test.afterEach(async ({ page }) => {
  // cleanup, e.g., logout or clear localStorage
  await page.evaluate(() => localStorage.clear());
});
```

- Fixtures: Playwright fixtures provide reusable setup (e.g., authenticated context). Create a fixture to perform login once and reuse `storageState` to speed tests.

```js
// example: use storageState from a login worker fixture
test.use({ storageState: 'storageState.json' });
```

- Tips:
  - Keep `beforeEach` fast; prefer API setup over UI flows when possible.
  - For expensive setup (seed DB), use `test.beforeAll()` with caution — it runs once per worker.

### Parallel & Cross-Browser Testing

- Playwright runs tests in parallel by default across worker processes. The `projects` array in `playwright.config.js` defines the browsers/devices to run (e.g., chromium, firefox, webkit). Each project runs the entire test suite unless filtered.
- Run a single browser locally:

```
npx playwright test --project=chromium
```

- Considerations for parallel tests:
  - Tests must be isolated: avoid shared global state (files, DB rows) unless you implement test-specific isolation (unique test data or test-specific DB namespaces).
  - Use unique ports or ephemeral resources when starting services per worker.
  - If many tests talk to the same backend, ensure the backend supports concurrent requests or run multiple backend instances.

- Cross-browser testing tips:
  - Use `projects` to define browser-specific options (e.g., device emulation).
  - Mark known browser-specific failures with `test.skip()` or `test.fixme()` and add annotations for tracking.
  - Use `--project` to narrow runs during development and run full matrix in CI.

### HTML Reporting

- Playwright's `html` reporter creates an interactive report including test results, traces, screenshots, and attachments. Configure reporter in `playwright.config.js`:

```js
reporter: 'html',
```

- Open the report locally after a run:

```
npx playwright show-report
```

- Advanced reporter config: you can combine reporters, e.g. `reporter: [['list'], ['html', { open: 'never' }]]` and then publish the generated `playwright-report` folder as CI artifacts.
- Artifacts included in the HTML report:
  - Screenshots (on failure)
  - Video recordings (if enabled in `use.video`)
  - Trace viewer links (if traces were recorded)

- CI usage:
  - Generate the report artifact and upload it in your CI pipeline for later inspection.
  - In GitHub Actions, upload the entire `playwright-report` folder as an artifact when tests finish.


## API testing with Playwright

Playwright also supports API testing through the `request` fixture. Example:

```js
const { request } = await playwright.request.newContext();
const resp = await request.get('/api/health');
expect(resp.ok()).toBeTruthy();
```

## Network mocking and fixtures

- Use `page.route()` to mock network requests for deterministic tests.
- Use fixtures for common setup (e.g., create test user via API before hitting UI tests).

## CI recommendations

- Ensure `npx playwright install --with-deps` (or equivalent) runs in CI to install browser dependencies.
- Use a stable Node version and cache `node_modules` between runs for speed.
- Use `--reporter=dot` or keep `html` and upload the HTML report artifacts for investigation.
- On CI set `CI=true` to enable CI-specific behavior (this repo uses `process.env.CI` in config).

Example GitHub Actions snippet (minimal):

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with: { node-version: '18' }
      - run: npm ci
      - run: npx playwright install --with-deps
      - run: npx playwright test --reporter=dot
      - run: npx playwright show-report --output=report || true
```

## Common pitfalls and tips

- Flaky tests: increase timeouts or add proper waits using locators; prefer `await locator.waitFor()` if necessary.
- Local server: use `webServer` in config to let Playwright manage starting/stopping the backend.
- Use `reuseExistingServer: true` when developing locally to speed up runs.
